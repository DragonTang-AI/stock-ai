### ⚠️ 五个 AI 的"糟粕"（我建议谨慎参考）



|建议点|我的判断|
|---|---|
|**Claude 版：Kafka \+ Milvus 全上**|太重了。V1 没有百万用户，Kafka 的维护成本远高于收益，行情数据直接 Redis Stream 就够了。Milvus 向量库同理——V1 场景下 LLM 直接 prompt 就能分析，不需要 RAG 检索历史研报|
|**LongCat 版：5 个数据库（PG \+ InfluxDB \+ MongoDB \+ Redis \+ Milvus）**|运维噩梦。V1 只需要 PostgreSQL \+ Redis，时序数据直接存 PG 的 TimescaleDB 扩展即可，一个 PG 实例搞定|
|**千问版：NestJS 做网关**|加一层 Node\.js 高并发网关表面好看，但 FastAPI 完全能扛 V1 的并发量，加 NestJS 只是增加团队维护负担。V2 用户量起来再用 Nginx/Kong 分流就够了|
|**Gemini 版：Ruflo 做 AI 司令部**|Ruflo 确实很酷，但它是个非常新的开源项目，文档和生态都不稳定。V1 用 LangGraph 做 Agent 编排更稳妥|
|**元宝版：直接绑定 Qlib 选股**|Qlib 是优秀的量化框架，但它的安装和数据初始化极其繁琐（数 GB 的 cn\_data），且模型偏传统 ML。V1 先用多 Agent \+ LLM 做选股分析，后续再接入 Qlib 做回测验证更灵活|

#### 🎯 这个融合版的关键设计决策



**1\. 统一Python技术栈（取Claude的架构理念 \+ 去千问的语言复杂度）**



第一期**不要**引入Node\.js网关 \+ Python业务层这种混合栈。一个人的项目或小团队项目，语言切换的认知成本远大于性能收益。FastAPI \+ Uvicorn 本身就能扛住较高并发，Nginx做反向代理就够了。



**2\. AI双引擎：Qlib\(量化信号\) \+ LangGraph多Agent\(LLM分析\)（取Gemini\+Claude之长）**



- **Qlib** 负责「数学层面」的选股信号——跑Alpha158因子、LightGBM模型，输出每日全市场评分排序，选出Top N

- **LangGraph多Agent** 负责「逻辑层面」的分析——技术面Agent看K线指标，基本面Agent看财报，舆情Agent看新闻，投委会Agent汇总加权输出置信分数 \+ 买/卖/持建议 \+ 自然语言文案

- 两者**互补但不耦合**：Qlib输出评分信号给投委会Agent作为参考维度之一，投委会再综合其他维度输出最终决策

    

**3\. 交易执行层轻量化（取Gemini的务实 \+ 去Claude的过度设计）**



第一期不要上Kafka。Redis Streams \+ Celery 对于日级别选股\+非高频交易场景完全够用。Kafka是大规模分布式团队协作时才需要的。



**4\. 数据库策略「够用即可，预留扩展」（取Claude的远见 \+ 去千问的堆砌）**



|数据类别|第一期方案|第二期后扩展|
|---|---|---|
|用户/账户/订单|PostgreSQL|保持不变|
|缓存/实时行情|Redis|Redis Cluster|
|K线/时序数据|**先用PG \+ 分区表**|数据量上来后迁移到 TimescaleDB|
|AI分析报告|PostgreSQL JSON字段|数据量上来后迁移到 MongoDB|
|向量知识库|第一期跳过|V2引入 Milvus 做 RAG|



> **核心原则**：不在第一期引入你还没遇到性能问题的数据库。PostgreSQL\+JSONB 存百万级分析报告完全够用。

#### 🔑 关键原则



1. **不要只用Cursor或只用Claude Code** — 两者是互补的

2. **CLAUDE\.md\(或\.cursorrules\) 是命脉** — 把所有架构约定、技术栈、命名规范写进去，让AI不跑偏

3. **小步提交** — 每次改不超过3\-5个文件，改完就测试\+commit

4. **测试优先** — 让AI先写pytest再写业务代码，防止静默错误

### 核心结论：最合适的二开基座



**首选方案：VeighNa \(原 Vn\.py\) \+ Qlib 混合架构**



没有单一的开源项目能完美覆盖你所有的需求（特别是“AI智能选股”\+“实盘交易接口”\+“移动端适配”）。最科学、最简单且符合国内生态的路径是 **“交易引擎用 VeighNa，AI 引擎用 Qlib”**。



#### 为什么是这个组合？



|需求模块|推荐组件|理由|
|---|---|---|
|**1\. 交易执行与券商对接**|**VeighNa \(Vn\.py\)**|**国内唯一事实标准**。原生支持 A 股（通过 XTP、中泰、华泰等接口）和港股（通过富途、盈透 IB）。它解决了最难的“账户系统打通”和“实盘稳定性”问题。|
|**2\. AI 选股与策略生成**|**Qlib \(微软\)**|**AI 量化最强库**。提供完整的机器学习工作流（数据处理\-\>模型训练\-\>回测）。你可以用 Qlib 训练出“每日推荐3\-5只股票”的模型，并输出置信度分数。|
|**3\. 模拟交易引擎**|**VeighNa 内置模拟器**|VeighNa 自带高精度的本地回测和模拟交易引擎，可以完美实现“初始资金、真实撮合逻辑”的模拟盘功能。|
|**4\. 数据看板与后端**|**FastAPI \+ Vue/Uni\-app**|VeighNa 是 Python 写的，天然适合用 FastAPI 封装成后端接口。前端用 Uni\-app 开发小程序/App，一套代码多端发布。|

#### 2\. 各功能模块实现方案



|功能需求|实现方案|关键技术点|
|---|---|---|
|**1\. 每日行情 AI 播报**|**Qlib \+ LLM**|用 Qlib 提取当日市场因子，结合 LLM（如通义千问 API）生成自然语言播报。|
|**2\. 每日选股推荐**|**Qlib 模型推理**|训练一个 Ranking 模型，每日收盘后运行推理，输出 Top 5 股票及预测收益率。|
|**3\. 持仓分析与置信分**|**自定义评分算法**|结合 Qlib 的预测概率 \+ 技术面指标（RSI/MACD）计算置信分（0\-100）。\<br\>0\-30: 抛, 30\-70: 持, 70\-100: 加。|
|**4\. 模拟交易 \(AI托管/手动\)**|**VeighNa 模拟网关**|**AI托管**：编写一个 VeighNa 策略类，直接接收 Qlib 的信号自动下单。\<br\>**手动**：前端发指令到后端，后端调用 VeighNa 的模拟接口下单。|
|**5\. 实盘交易 \(AI托管/手动\)**|**VeighNa 实盘网关**|**关键**：需要申请券商的量化接口（如中泰 XTP、华泰 MMT）。\<br\>**AI托管**：同模拟盘，但连接实盘网关。\<br\>**手动**：前端指令 \-\> 后端风控检查 \-\> 实盘网关下单。|
|**6\. 券商账户打通**|**VeighNa Gateway**|VeighNa 已封装好主流券商接口。你需要做的是：\<br\>1\. 申请券商量化权限。\<br\>2\. 在 VeighNa 中配置账号密码/API Key。|





### 为什么不建议其他项目？



1. **Backtrader / VectorBT**：

    - **缺点**：它们主要是**回测框架**，缺乏原生的、稳定的**实盘交易网关**。你要自己写对接券商的代码，难度极大且不稳定。

    - **适用**：仅适合做纯研究，不适合做带实盘交易的 App。

        

2. **FinRL**：

    - **缺点**：专注于强化学习算法研究，工程化程度低，没有现成的交易接口和账户管理系统。你需要从零搭建交易系统。

    - **适用**：适合学术研究，不适合快速产品化。

        

3. **QuantConnect / Lean**：

    - **缺点**：主要面向美股/全球市场，对 A 股/港股的支持不如 VeighNa 深入，且社区资源在国内较少。

### 第一期 MVP（最小可行性产品）开发建议



为了“最简单、最科学”地达到目的，建议分步走：



#### 阶段一：数据与 AI 核心（2\-3周）

1. **数据源**：使用 **AkShare**（免费、开源、支持 A股/港股）获取历史行情。

2. **AI 模型**：基于 **Qlib** 搭建一个简单的 LightGBM 或 LSTM 模型，训练 A 股数据。

关键决策点 — 我的判断

1. 要不要上微服务？

第一版不要。5份架构里Claude版明确说了微服务过度设计是坑。你现在的团队规模（solo或小团队），微服务的分布式事务、服务发现、链路追踪会吃掉你50%的开发时间。



做法：用FastAPI的APIRouter做模块化隔离，每个模块（user/trade/analysis/report）独立目录，独立数据库schema。未来需要拆分时，把目录拎出来就是独立服务。



2. Python统一栈 vs Node\.js网关\+Python后端？

统一Python（FastAPI）。千问说的Node\.js网关处理高并发有道理，但代价是团队要维护两套技术栈。FastAPI基于Starlette，异步性能足够应对第一版。真到了需要拆分网关的那天，加一层Nginx \+ Lua或Envoy即可。



3. 多智能体应该用哪种模式？

Gemini的"投委会辩论"模式最优。理由：



LongCat和千问的智能体方案太抽象，没有给出具体的协调机制

Claude版和元宝版根本没涉及多智能体

Gemini的「技术面/基本面/舆情面Agent分头分析 → 投委会Agent加权决策」最贴近真实投资流程

实现方式：用LangGraph编排，每个Agent通过LLM调用\+工具调用（查数据）完成分析，输出结构化JSON，投委会做最终融合。这个模式可解释性强，用户能看到每个Agent的"思考过程"。



4. Qlib vs 自研选股？

Qlib作为基础 \+ 上层用LLM增强。Qlib的Alpha158因子集和LightGBM模型是经过市场验证的，不需要自己从头搞因子工程。但从Qlib输出Top5后，用LLM做二次筛选和可解释性生成，这样既有量化模型的严谨，又有LLM的自然语言表达能力。



5. TimescaleDB vs InfluxDB？

TimescaleDB。因为它是PostgreSQL扩展而非独立数据库，跟业务数据在同一个生态里，用SQL查询即可，不需要学新的查询语言。当你说"查过去3个月的每日收盘价"时，TimescaleDB的hypertable比InfluxDB更适合关系型思维。



6. Kafka有必要吗？

第一版不需要。Claude版的Kafka设计很科学，但那是给百万用户、开盘瞬间千万级数据量准备的。第一版直接用Redis Pub/Sub \+ Celery的定时任务完全够用。等用户量上来再引入Kafka。



7. Milvus向量数据库？

V2再考虑。LongCat和Claude版都提了Milvus做RAG，但第一版你还没有积累足够的研报/新闻数据。先把基础功能跑通，有数据沉淀后再上RAG增强。

### 二、 取其精华，去其糟粕：文档关键点修正

在仔细阅读你的两份文档后，我认为需要对“千问版”和“gemini版”中几处**不符合国内金融实盘现状的硬伤**进行修正（去其糟粕）：

1. **关于实盘打通（避坑重点）**：

    - *去其糟粕*：千问版提到的“Qbot”以及Gemini版提到的“裸爬同花顺/东财网页”在国内 2026 年是不可行的。国内券商对 Web/App 逆向的自动化脚本封杀极其严格，极易导致封号，且有法律风险。

    - *取其精华*：元宝版和Claude版提出的 **MiniQMT \(通过 XtQuant 库本地驱动券商终端\) 才是个人和solo独立开发者目前唯一合规、稳定、免费的 A 股实盘自动化下单解法**。港股则老老实实走**富途 OpenAPI**。

2. **数据层架构简化**：

    - *去其糟粕*：Claude 版提到直接引入大型 Kafka 消息队列。对于 V1 阶段的单兵/小团队开发来说，Kafka 的运维成本过高，容易卡死在基础设施配置上。

    - *取其精华*：第一期使用 **Redis 的 Stream 或 Pub/Sub 功能** 替代 Kafka 充当行情缓冲队列。Redis 具备 10w\+ 级别的单秒处理能力，足够应付 A 股和港股的 K 线、Tick 推送，后续用户量过百万再平滑重构到 Kafka。

3. **多 Agent 协作的定位**：

    - *修正*：不能让大模型直接面对每天数千只股票。大模型的算力、Token 价格和推理速度决定了它不能做“海选”。**必须由 Qlib 模型利用数学指标从 5000 只股票中筛选出前 20\-30 只（初选），然后再把这 30 只股票丢给 Ruflo 多智能体矩阵，让 Agent 去看它的新闻、财报、研报进行精选和拟写报告（精选）**。这样既省钱，又科学。
